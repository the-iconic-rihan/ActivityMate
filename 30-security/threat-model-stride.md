# Threat Model — STRIDE Analysis

## Overview

This document is a STRIDE (Spoofing, Tampering, Repudiation, Information Disclosure, Denial of Service, Elevation of Privilege) threat model scoped specifically to ActivityMate's actual risk surface, not a generic checklist. ActivityMate's distinguishing risk factors, versus a typical consumer app, are: (1) it verifies real-world identity to build trust between strangers, making identity spoofing high-value to attack; (2) it handles precise geolocation tied to real-world meetups, making location disclosure a physical-safety issue, not just a privacy one; (3) its core value (activity discovery, trust scores) is built from user-generated data that is attractive to scrape at volume for competitor cloning or targeted harassment; (4) an account takeover here doesn't just expose data, it lets an attacker impersonate a "trusted" verified person to a stranger who has agreed to meet them in person.

## Context / Problem

Applying a generic STRIDE checklist (SQL injection here, CSRF there) would produce a document that is true of almost any web application and does not help engineering prioritize. The goal instead is to walk each STRIDE category through ActivityMate's specific architecture — auth ([12-auth](../12-auth/README.md)), location ([17-maps-location](../17-maps-location/README.md)), trust/verification ([13-trust-safety-fraud](../13-trust-safety-fraud/README.md)) — and name the concrete attack and concrete mitigation for each.

## Options Considered

1. **Generic OWASP-style checklist covering common web vulnerabilities.** Rejected as the primary artifact — that content already lives in [secure-coding-standards.md](secure-coding-standards.md); duplicating it here would dilute focus on ActivityMate-specific risks.
2. **Asset-centric threat model** (model threats per data asset: profile, location, chat, trust score). Considered — useful lens, but STRIDE-per-category was selected as the primary structure because it maps more directly to the mitigation owners (auth team owns spoofing controls, infra owns DoS, etc.) and is the format named in the folder README.
3. **STRIDE-per-category, each row naming a concrete ActivityMate scenario and mitigation, with asset context folded into each row.** Selected.

## STRIDE Threat Table

| Category | Concrete ActivityMate Risk | Attack Scenario | Mitigation | Owner |
|---|---|---|---|---|
| **Spoofing** | Fake / bot profiles bypassing verification | An attacker scripts account creation with stolen or synthetic phone numbers and stock-photo selfies to farm accounts that appear "phone-verified" but are not a real accountable person, then uses them to scam or harass real users who trust the verification badge | Tiered verification (phone, email, selfie liveness check, optional government ID/LinkedIn per [PRD-001](../PRD-001_Master_Product_Requirements.md) §9) with liveness detection (not just a static photo match) on the selfie step per [../12-auth/identity-verification-kyc.md](../12-auth/identity-verification-kyc.md); phone verification tied to SMS OTP with carrier-level fraud signals where available; rate-limiting and device-fingerprinting on account creation to blunt scripted mass signup; trust score starts low and unverified/low-trust accounts are constrained (see [../13-trust-safety-fraud/user-verification-levels.md](../13-trust-safety-fraud/user-verification-levels.md)) from initiating activities until they clear verification | BE / Trust & Safety |
| **Spoofing** | Session/token theft leading to identity impersonation | Stolen JWT/session token (via XSS, insecure storage on a rooted device, or network interception) lets an attacker act as a verified user, including messaging that user's matched activity partners | Short-lived access tokens with refresh rotation, secure token storage on-device (Flutter secure storage / Keychain-equivalent), token binding to device fingerprint where feasible, and forced re-auth for sensitive actions (viewing government ID status, changing phone number) per [../12-auth/session-token-management.md](../12-auth/session-token-management.md) | BE / Security |
| **Tampering** | Trust score or rating manipulation | An attacker creates sockpuppet accounts to inflate their own trust score with fake positive ratings, or coordinates brigading to tank a rival's/victim's score | Trust score inputs (completed activities, ratings, reports, cancellation rate, response rate, account age per PRD §9) are computed server-side only, never client-submitted directly; rating events are tied to a verified completed-activity record (no rating without a corroborated check-in on both sides) per [../13-trust-safety-fraud/trust-engine-architecture.md](../13-trust-safety-fraud/trust-engine-architecture.md); anomaly detection flags rating velocity/clustering consistent with sockpuppet rings, per [../13-trust-safety-fraud/fraud-detection-system.md](../13-trust-safety-fraud/fraud-detection-system.md) | BE / Trust & Safety |
| **Tampering** | Client-side location spoofing (GPS spoofing apps) | A user fakes their GPS location to appear near a desirable activity, or to fake a check-in at a meetup point they never physically attended, undermining the "confirmed attendance" safety signal | Check-in requires proximity confirmation with sanity checks (implausible speed/jump detection between location updates), and mock-location/developer-option detection on Android where the OS exposes it; check-in state is corroborated where possible against the activity's declared meetup point per [../17-maps-location/meetup-point-safety-recommendations.md](../17-maps-location/meetup-point-safety-recommendations.md) — treated as a strong signal, not cryptographic proof, since GPS spoofing can never be fully eliminated client-side | BE / Location |
| **Repudiation** | A user denies sending an abusive/threatening chat message, or denies having reported (or not reported) a safety incident | In a dispute or safety investigation, there is no reliable record of who said what, undermining moderation and any law-enforcement request under [../31-privacy-compliance-legal/law-enforcement-request-policy.md](../31-privacy-compliance-legal/law-enforcement-request-policy.md) | All chat messages are persisted server-side (not solely client-side/ephemeral) with sender identity, timestamp, and immutable audit trail; message edits/deletes are soft-deleted (hidden from the other party if desired, retained for moderation/legal per [../31-privacy-compliance-legal/data-retention-deletion-policy.md](../31-privacy-compliance-legal/data-retention-deletion-policy.md)); all moderation actions and report submissions are logged in an admin audit log per [../24-admin-panel/audit-log-spec.md](../24-admin-panel/audit-log-spec.md) | BE / Trust & Safety |
| **Information Disclosure** | Location-based stalking via inferred precise location | A malicious matched user (or an attacker who compromises an account) repeatedly observes another user's activity check-ins or "near me" discovery results to triangulate their home, workplace, or daily routine — the single highest-severity privacy risk specific to this product category | Location is fuzzed/rounded for discovery and profile display (never exact coordinates shown to other users) per [../17-maps-location/location-privacy-strategy.md](../17-maps-location/location-privacy-strategy.md); precise location is only ever used server-side for check-in proximity verification and is not retained longer than needed (see [../31-privacy-compliance-legal/data-retention-deletion-policy.md](../31-privacy-compliance-legal/data-retention-deletion-policy.md)); repeated proximity queries against the same target user are rate-limited and pattern-detected as potential stalking behavior | BE / Location / Trust & Safety |
| **Information Disclosure** | Bulk scraping of user profiles, location patterns, and activity data | A competitor, data broker, or malicious actor scripts systematic scraping of public-facing profile and activity-discovery endpoints to build a shadow database of users' patterns, undermining both competitive position and user privacy at scale | API-level rate limiting and anomaly detection on discovery/search endpoints (Meilisearch-backed) per [../08-backend-services/rate-limiting-standards.md](../08-backend-services/rate-limiting-standards.md); pagination limits and no bulk-export endpoints exposed publicly; CAPTCHA or equivalent challenge on anomalous request patterns; monitoring for scraping-signature traffic (high-volume sequential ID enumeration, unusual geographic query fan-out) | SEC / BE |
| **Information Disclosure** | Government ID / selfie image leak | A misconfigured storage bucket, insecure direct object reference (IDOR) on the verification-document endpoint, or an insider with excessive access exposes users' government ID scans — a DPDP Act "significant harm" category of personal data | MinIO buckets are private with signed, short-lived, per-request URLs only (see [../18-media-storage/storage-cdn-strategy.md](../18-media-storage/storage-cdn-strategy.md)); access to verification documents is restricted to the verification service and a narrow, logged admin role (see [../24-admin-panel/rbac-admin-roles.md](../24-admin-panel/rbac-admin-roles.md)); documents are deleted per the retention schedule once verification is complete rather than retained indefinitely (see [../31-privacy-compliance-legal/data-retention-deletion-policy.md](../31-privacy-compliance-legal/data-retention-deletion-policy.md)) | SEC / BE / Legal |
| **Denial of Service** | Targeted DoS against the panic-button or check-in service during a live meetup window | An attacker (or simply organic Friday-night peak load) degrades or takes down the safety-critical path exactly when it is most likely to be needed | Panic button and check-in are isolated with dedicated capacity/priority (not competing with, e.g., bulk image upload traffic) per [../07-architecture/scalability-plan-0-to-10m.md](../07-architecture/scalability-plan-0-to-10m.md); rate limiting protects against abuse without throttling legitimate safety-critical calls; load-tested explicitly under peak-weekend simulation per [../29-testing-qa/load-performance-testing-plan.md](../29-testing-qa/load-performance-testing-plan.md); SLOs and paging defined in [../28-monitoring-observability/sla-slo-definitions.md](../28-monitoring-observability/sla-slo-definitions.md) | DO / BE |
| **Denial of Service** | Application-layer DoS via verification pipeline abuse | An attacker submits a high volume of selfie/ID verification requests to exhaust third-party verification vendor quota/cost or backend processing capacity | Rate limiting and per-account/per-device caps on verification attempts; CAPTCHA on repeated failed attempts; cost-alerting on third-party verification vendor spend per [../25-infrastructure-cloud/cost-optimization-playbook.md](../25-infrastructure-cloud/cost-optimization-playbook.md) | BE / DO |
| **Elevation of Privilege** | Account takeover leading to trust-badge impersonation | An attacker takes over a real, previously-verified user's account (credential stuffing, SIM swap, phishing) and now appears to strangers as a trusted, high-trust-score individual — the specific danger being that the trust signal this platform is built on becomes a weapon in the attacker's hands for an in-person meetup | Strong auth (OTP + optional second factor), anomaly detection on login (new device/location triggers step-up verification), session invalidation on password/phone-number change, and a user-facing "recent activity" view so users can detect takeover per [../12-auth/account-recovery-security.md](../12-auth/account-recovery-security.md); trust score is suppressed/flagged pending re-verification if account-takeover indicators are detected | SEC / BE |
| **Elevation of Privilege** | Regular user gaining admin/moderator panel access | A privilege-escalation bug in the RBAC implementation, or a leaked admin credential, lets a non-staff user reach moderation tools — e.g., to clear their own reports or view other users' verification documents | Strict RBAC enforcement server-side on every admin endpoint (never trust a client-supplied role claim) per [../12-auth/authorization-rbac-model.md](../12-auth/authorization-rbac-model.md) and [../24-admin-panel/rbac-admin-roles.md](../24-admin-panel/rbac-admin-roles.md); admin panel requires separate, stronger authentication (not reachable via the consumer app's session) and is IP-allowlisted or VPN-gated where feasible; all admin actions are audit-logged per [../24-admin-panel/audit-log-spec.md](../24-admin-panel/audit-log-spec.md) | SEC / BE |

## Recommendation & Rationale

Mitigations above are prioritized by two factors: (1) physical-safety consequence — location disclosure and account-takeover-for-impersonation rank highest because failure translates to real-world harm, not just data exposure; (2) trust-model integrity — spoofing and tampering rows are weighted heavily because the entire product thesis depends on the trust score and verification badges being genuinely hard to fake. This table should be revisited each time a new MVP feature ships (matching, communities, payments) rather than treated as a one-time exercise — each is a new attack surface.
